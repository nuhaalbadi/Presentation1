› Introduction

eXtreme Programming (XP) is a software development life cycle and one of the most well-known agile processes. It is based on a number of values, activities, rules, and practices which will be discussed in following sections. Its main focus is on providing the customer with a high quality software in the shortest time possible. Instead of delivering one final version of the software, XP delivers the software in pieces as needed by the customer allowing for more interaction with the customer and more tolerance to change.

In fact, XP is not only tolerant to change, but also it advocates the concept of "Embracing Change". In XP, change is always a good thing. Change inevitable, customer requirements get clearer as we develop the system. It is hard to define the system completely from the start. Therefore, XP tries to reduce the cost of change as possible by incorporating simple well written designs/code and by anticipating change and designing for it in all its practices.

As shown in lecture 7, the name "Extreme" comes from the fact that XP takes certain software development practices and sets them to extreme levels. For example, testing in XP is done multiple times a day. When a change is made, all tests should be run again, extreme testing.

› History

In March, 1969 XP was first developed and used by Kent Beck who was brought to work on a project at Daimler Chrysler. He was trying to find better ways to enhance the software development process. His ideas and methodologies were proven to be successful which later became known as "Extreme Programming"

A short history of XP is best summarized as

"Kent Beck found himself dictating practice on a project for which he had assumed some personal risk. Unsure of which to emphasize more, he chose to "turn all the knobs to ten.""

In 1999, Kent wrote a book titled "Extreme Programming Explained Embrace Change" in which he explained in detail the XP methodology. The four main dimensions that Kent based his methodology around are communication, simplicity, feedback, and courage.

In 2000, another book titled "Extreme Programming Installed" came out that was written by Jeffries, Anderson, and Hendrickson who participated in the Daimler Chrysler's project, C3, which describes the XP methodology and what makes it work in detail.

› Concept

XP is built upon a number of activities, values, and rules, each is discussed in detail below.

› XP Activities

Planning is the first activity (phase) in the XP life cycle. Then, each iteration consists of four main activities: coding, testing, designing, and listening.

  • Planning 

    • The XP development life cycle starts by meeting with customers and developing "user stories", each user story describes a requirement of the system. These user stories are get prioritized by the customers which leads to developing a release plan. Then, the development team distribute these user stories into iterations. Each iteration will consist of user stories that sums up to the "project velocity", which is a mean of measuring how much work is done in each iteration. Work load for the coming iteration gets adjusted according to the most recent project velocity. No detailed requirements is done in this phase since requirements get clearer as we develop the software and get constant feedback from customers.

  • Designing

    • The designing process in XP should follow the following guidelines and principles:

    • Always seek simplicity in design. Simple design is faster and easier to implement and easier to change.

    • Use "system metaphor" when naming system classes, objects, etc. This helps maintain consistency and compatibility across the system. It also facilitates with the overall understanding of the system.

    • Use Class Responsibilities and Collaboration (CRC) cards to allow for more cooperation in the design process. All members can be part of the design process by sharing design ideas on such cards. CRC cards implements the object methodology in design as opposed to the traditional procedural one.

    • Create what is called "spike solutions" , these are simple programs that address complicated problems without worrying about any other aspect of the system. This technique helps reduce the risk of of some technical problems. It also helps with giving better estimates for user's story.

  • Coding

    • In XP, how much of the system is up and running in the customer’s site defines how much progress has been achieved. Therefore, coding is one of the most important phases in XP as it allows for producing something that the customer can actually use and give feedback on. For that, developers are discouraged to waist their time with documentations and other software artifacts as these will extremely slow the development time and the customers would have to wait a long time before they actually get something.

  • Testing

    • In XP, testing is an integral part of the development process. Testing is done as the code is being developed. The two main types of testing are

    • Unit testing: In order for a code to be released, it must pass all its unit tests. In, XP unit tests should be automated and developed for each class, function, etc. in the code. Whenever a change is made in the code, all unit tests should be run again using an automated test framework. Developers write these tests with the intern to "break" their code in order to find as much bugs as possible.

    • Acceptance testing: At each iteration, acceptance tests are generated for each of the selected user stories according to the customer specifications. At the end of each iteration, the customer is provided with the results of these tests and decides which tests are failed and which are passed. Acceptance tests insure that the system implements exactly what the customer wants.

  • Listening

    • Getting feedback from customers is a key activity in XP. XP encourages developers to be good listeners for customers. Getting feedback and welcoming changes helps developing a system that best match customer’s specifications. Feedback is not done only between the development team and customers; it is also encouraged between the development team themselves. A developer should give/seek feedback to/from other co-developers. Communication and feedback are highly valued in XP as you will see in the following section.

› XP Values

  • Simplicity

    • Team members should always seek simplicity, but efficiency, in their all practices. In XP, they only worry about things that are needed, they should not waist their time worrying about things they might need. This helps the team stay on track and deliver faster.

    Kent says in Extreme Programming Explained

    "XP is making a bet. It is betting that it is better to do a simple thing today and pay a little more tomorrow to change it if it needs it, than to do a more complicated thing today that may never be used anyway."

  • Communication

    • Team members should continuously communicate with each other and with costumers. Face to face communication is a key thing in XP. Everyone is involved in the development process most importantly customers.

  • Feedback

    • In XP you should always get feedback from customers and be ready to change frequently. This is very essential in XP since you deliver a working software to the customer at the end of every iteration. The customers actually use the software and give back feedback that it is extremely helpful for next iterations.

    XP exempts you from waiting until the end of the software development process to see whether or not the customer actually likes your software. In XP this is done starting from the early stages of the software development process. This means that you should expect change and embrace it. In XP change is easy to be done because the whole software is implemented in a way that allow changes to be made with the lowest cost possible.  

  • Courage

    • In XP you should be courageous enough to simply "throw away" any code that you think is not good enough. Do not worry about the time you spent writing this code because simply you can start fresh tomorrow and write a better one. The "throwaway" approach in XP applies also to design. Do not worry about coming up with right design from the first time, just try whatever is in your head and see where that gets you.

    Kent says in Extreme Programming Explained

    "Maybe you have three design alternatives. So, code a day's worth of each alternative, just to see how they feel. Toss the code and start over on the most promising design."

  • Respect

    • Respect is highly appreciated in XP. It should be practiced by every member of the team including customers and managers. No one should feel disrespected or discouraged. Respect helps keep the spirit of the team up and thus increasing their performance.

 

› Practices

Although there are 28 practices in XP, they can be summed up into the following 12 key practices:

  • User Stories (The Planning Game)

    The game has 3 main rules:

    • The customer specifies system functionalities which will be converted into brief "user stories" and will be written on an index card. What makes XP great is that stories can be added at any time during the development life cycle.

    • The development team gives general estimates on how long each story will take to develop. This estimate is usually given in terms of weeks, 1,2, or 3 weeks. If a story will take more than 3 weeks, the customer splits the story into two simpler stories. If a story will take less than a week, then the customer merges it with another story.

    • The customer prioritizes user stories according to their importance. Then, the customer and the development team decides on which stories will be implemented in the next iteration, specifying "project velocity."

  • Small Releases

    This is a key practice in XP. a new version of the software is delivered to the customer at the end of each iteration. Each version adds a small functionality to the software.

  • System Metaphor

    Having a System metaphor, as explained earlier, provides a naming conventions and gives a general architecture of the system. It makes it easier for developers to adhere to naming conventions, and it facilitates communication and understanding between team members.

  • Coding Standard

    Developers should use agreed-upon standards in coding styles. Although, this might be hard for developers to get use to, it has a great advantage of making code readable for all developers which encourages sharing and facilitates code refactoring.

  • Pair Programming

    One each work station, two developers work on the same code but each performing a different task. Although pair programming might slightly slow developers down, a study showed that it produces a higher quality code.

    "Laurie Williams of the University of Utah in Salt Lake City has shown that paired programmers are only 15% slower than two independent individual programmers, but produce 15% fewer bugs. Since testing and debugging are often many times more costly than initial programming, this is an impressive result."

  • Collective Code Ownership

    Code belongs to all developers, every one can modify, refactor, and add to code. By applying coding standard, collective code ownership becomes easier. Collective code ownership significantly reduces the risk of team turnover because all developers contribute in somehow to each part of the system.
