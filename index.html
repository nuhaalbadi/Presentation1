› Introduction

eXtreme Programming (XP) is a software development life cycle and one of the most well-known agile processes. It is based on a number of values, activities, rules, and practices which will be discussed in following sections. Its main focus is on providing the customer with a high quality software in the shortest time possible. Instead of delivering one final version of the software at the end of the development process, XP delivers the software in pieces as needed by the customer allowing for more interaction with the customer and more tolerance to change.

In fact, XP is not only tolerant to change, but also advocates the concept of "Embracing Change." In XP, change is always a good thing. Change is inevitable, customer requirements get clearer as the system is being developed. It is hard to define the system completely from the start. Therefore, XP tries to reduce the cost of change as much as possible by incorporating simple well written designs/code and by anticipating change and designing for it in all its practices.

As shown in lecture 7, the name "Extreme" comes from the fact that XP takes certain software development practices and sets them to extreme levels. For example, testing in XP is done multiple times a day. When a change is made, all tests should run again, extreme testing.

› History

In March, 1969 XP was first developed and used by Kent Beck who was brought to work on a project at Daimler Chrysler. He was trying to find better ways to enhance the software development process. His ideas and methodologies were proven to be successful which later became known as "Extreme Programming."

A short history of XP is best summarized as

"Kent Beck found himself dictating practice on a project for which he had assumed some personal risk. Unsure of which to emphasize more, he chose to "turn all the knobs to ten.""

In 1999, Kent wrote a book titled "Extreme Programming Explained Embrace Change" in which he explained in detail the XP methodology. The four main dimensions that Kent based his methodology around are communication, simplicity, feedback, and courage.

In 2000, another book titled "Extreme Programming Installed" came out that was written by Jeffries, Anderson, and Hendrickson who participated in the Daimler Chrysler's project, C3, which describes the XP methodology and what makes it work in detail.

› Concept

XP is built upon a number of activities, values, and rules, each is discussed in detail below.

› XP Activities

Planning is the first activity (phase) in the XP life cycle. Then, each iteration consists of four main activities: coding, testing, designing, and listening.

  • Planning 

    • The XP development life cycle starts by meeting with customers and developing "user stories", each user story describes a requirement of the system. These user stories get prioritized by the customer which leads to developing a release plan. Then, the development team distributes these user stories into iterations. Each iteration will consist of user stories that sums up to the "project velocity", which is a mean of measuring how much work is done in each iteration. Work load for the coming iteration gets adjusted according to the most recent project velocity. No detailed requirements is done in this phase since requirements get clearer as we develop the software and get constant feedback from customers.

  • Designing

    • The designing process in XP should follow the following guidelines and principles:

    • Always seek simplicity in design. Simple design is faster, easier to implement, and easier to change.

    • Use "system metaphor" when naming system classes, objects, etc. This helps maintain consistency and compatibility across the system. It also facilitates with the overall understanding of the system.

    • Use Class Responsibilities and Collaboration (CRC) cards to allow for more cooperation in the design process. All members can be part of the design process by sharing design ideas on such cards. CRC cards implement the object methodology in design as opposed to the traditional procedural one.

    • Create what is called "spike solutions," these are simple programs that address complicated problems without worrying about any other aspect of the system. This technique helps reduce the risk of of some technical problems. It also helps with giving better estimates for user's story.

  • Coding

    • In XP, how much of the system is up and running in the customer’s site defines how much progress has been achieved. Therefore, coding is one of the most important phases in XP as it allows for producing something that the customer can actually use and give feedback on. For that, developers are discouraged to waist their time with documentations and other software artifacts as these will extremely slow the development time and the customers would have to wait a long time before they actually get something in their hand.

  • Testing

    • In XP, testing is an integral part of the development process. Testing is done as the code is being developed. The two main types of testing are

    • Unit testing: In order for a code to be released, it must pass all its unit tests. In, XP unit tests should be automated and developed for each class, function, etc. in the code. Whenever a change is made in the code, all unit tests should run again using some automated test frameworks. Developers write these tests with the intention to "break" their code in order to find as much bugs as possible.

    • Acceptance testing: At each iteration, acceptance tests are generated for each of the selected user stories according to the customer specifications. At the end of each iteration, the customer is provided with the results of these tests and decides which tests are failed and which are passed. Acceptance testing insures that the system implements exactly what the customer wants.

  • Listening

    • Getting feedback from customers is a key activity in XP. XP encourages developers to be good listeners for customers. Getting feedback and welcoming changes helps develop a system that best matches customer’s specifications. Feedback is not done only between the development team and customers; it is also encouraged between the development team themselves. A developer should give/seek feedback to/from other co-developers. Communication and feedback are highly valued in XP as you will see in the following section.

› XP Values

  • Simplicity

    • Team members should always seek simplicity, but efficiency, in their all practices. In XP, they only worry about things that are needed, they should not waist their time worrying about things they might need. This helps the team stay on track and deliver faster.

    Kent says in Extreme Programming Explained

    "XP is making a bet. It is betting that it is better to do a simple thing today and pay a little more tomorrow to change it if it needs it, than to do a more complicated thing today that may never be used anyway."

  • Communication

    • Team members should continuously communicate with each other and with costumers. Face to face communication is a key thing in XP. Everyone is involved in the development process, most importantly customers.

  • Feedback

    • In XP you should always get feedback from customers and be ready to change frequently. This is very essential in XP since you deliver a working software to the customer at the end of every iteration. The customers actually use the software and give back feedback that it is extremely helpful for next iterations.

    XP exempts you from waiting until the end of the software development process to see whether or not the customer actually likes the software. In XP this is done starting from the early stages of the software development process. This means that you should expect change and embrace it. In XP change is easy to be done because the whole software is implemented in a way that allow changes to be made with the lowest cost possible.  

  • Courage

    • In XP you should be courageous enough to simply "throw away" any code that you think is not good enough. Do not worry about the time you spent writing this code because simply you can start fresh tomorrow and write a better one. The "throwaway" approach in XP applies also to design. Do not worry about coming up with right design from the first time, just try whatever is in your head and see where that gets you.

    Kent says in Extreme Programming Explained

    "Maybe you have three design alternatives. So, code a day's worth of each alternative, just to see how they feel. Toss the code and start over on the most promising design."

  • Respect

    • Respect is highly appreciated in XP. It should be practiced by every member of the team including customers and managers. No one should feel disrespected or discouraged. Respect helps keep the spirit of the team up, and thus increasing their performance.

 

› Practices

Although there are 28 practices in XP, they can be summed up into the following 12 key practices:

  • User Stories (The Planning Game)

    The game has 3 main rules:

    • The customer specifies system functionalities which will be converted into brief "user stories" and will be written on an index card. What makes XP great is that stories can be added at any time during the development life cycle.

    • The development team gives general estimates on how long each story will take to develop. These estimates are usually given in terms of weeks, 1,2, or 3 weeks. If a story is estimated to take more than 3 weeks, the customer splits the story into two simpler stories. If a story is estimated to take less than a week, then the customer merges it with another story.

    • The customer prioritizes user stories according to their importance. Then, the customer and the development team decides which stories to be implemented in the next iteration, specifying "project velocity."

  • Small Releases

    This is a key practice in XP. a new version of the software is delivered to the customer at the end of each iteration. Each version adds a small functionality to the software.

  • System Metaphor

    Having a System metaphor, as explained earlier, provides a naming conventions and gives a general architecture of the system. It makes it easier for developers to adhere to naming conventions, and it facilitates communication and understanding between team members.

  • Coding Standard

    Developers should use agreed-upon standards in coding styles. Although, this might be hard for developers to get use to, it has a great advantage of making code readable for all developers which encourages sharing and facilitates code refactoring.

  • Pair Programming

    On each work station, two developers work on the same code but each perform a different task. Although pair programming might slightly slow developers down, a study showes that it produces a higher quality code.

    "Laurie Williams of the University of Utah in Salt Lake City has shown that paired programmers are only 15% slower than two independent individual programmers, but produce 15% fewer bugs. Since testing and debugging are often many times more costly than initial programming, this is an impressive result."

  • Collective Code Ownership

    Code belongs to all developers, every one can modify, refactor, and add to code. By applying coding standards, collective code ownership becomes easier. Collective code ownership significantly reduces the risk of team turnover because all developers contribute in somehow to each part of the system.

  • Testing

    As mentioned earlier, there are two main categories of testing in XP, unit testing and acceptance testing. When a new feature is to be added to the system, you should write a unit test for it first, then you write a code that passes that test. This methodology of testing is called Test-Driven Development. Writing tests first makes writing code easier because that way you know exactly what you need to do, i.e. write a code that passes the tests. Also, when you write tests first, you always know when you have done coding all functionalities, when all tests are passed.

  • Continuous Integration

    Whenever a software build is ready to be released, it needs to be integrated to the current system so that everyone works on the latest version of the system. This integration is done multiple times a day, current practices reduced this requirement to be once a week. Continuous integration has the advantage of discovering integration problems early, which leads to easier and less fix cost.

  • Simple Design

    There are three main rules that developers need to follow regarding the "simple design" practice:

    • Choose the simplest design that works.

    • Do not design the same functionality twice, reuse your design.

    • Do not waist your time designing functionalities that you do not need or ones that are scheduled for future iterations.

    As a rule, the simpler the design, the better the system, and the more it becomes tolerant to change.

  • Refactoring

    Refactoring code simply means improving it. This improvement can be done by removing duplicate code, removing unused functionalities, or redesigning the code to make it simpler. All team members are encouraged to refactor their own code as well as code written by other co-developers, collective ownership. Refactoring keeps the system simple, concise and hence easy to understand and modify. The testing mechanism in XP makes refactoring done without the overhead of worrying about breaking the code. This is because after making any change, you will always have to run all tests again and know if something is not working and fix it.  

  • 40-Hour Workweek

    In XP, there are two main types of planning:

    Release planning: Release planning results in creating a release plan which defines the overall scope and schedule for the project.

    Iteration planning: Iteration planning is done at the beginning of each iteration which defines what stories "features" of the system should be implemented during each iteration.

    The good thing about XP, and agile methods in general, is that it is okay to be wrong in your original estimates. The plans mentioned above are not fixed, they can and should be adjusted according to the follow of work. Therefore, developers should never work overtime. Working overtime leads to developers being exhausted which in turns leads to writing code with more bugs and less quality. In XP, when things are not going as planned, call for an iteration planning meeting and modify the project velocity so that developers can focus now on fewer things and get things done by the end of the iteration. If it takes, call for a release planning meeting and change the whole scope of the project or extend the project timing. No matter what happens, in XP you can never break the rule of "40-hour work week."

  • On-Site Customer

    In XP, the customer is an essential part of the development team. One of the XP's core values is face-to-face communication, therefore, the customer needs to be available on-site to communicate with the development team and and to answer questions. Recall that user stories are brief abstract statements that describes system's functionalities. Therefore, the development team may need to talk to the customer and get more details on a user story to make sue that they understand it fully. Also, the customer should be present during meetings to negotiate and help in making decisions.

› Pros & Cons

Like any other software development life cycle, XP has its ups and downs. Whether you should use XP or not depends on a number of factors such as the type of project being developed, and the size of the development team. The following lists of Advantages and disadvantages of XP helps you make that choice.

  • Advantages of XP

    • XP embraces change and designs for it. Therefore, requirements need not to be specified in advance, which gives a great flexibility to customers who do not know exactly what they want.

    • XP lowers the risk of building new challenging projects that have high risks, and increases their success rate.

    • Since XP is all about communication, it works best when you have a small development team, typically between 2 and 12 members.

    • Involving the customer is extremely helpful and leads to creating a software that matches what the customer really wants.

    • XP produces a software with less bugs and defects because it is continuously tested and intergraded. Also, in XP code is written in pairs and can be viewed and modified by all the team.

    • XP delivers the software faster to the customer. The customer can get a working software as early as the first few weeks from the start of the project.

    • The feedback from customers helps to detect problems with the system early and hence can be fixed with low cost.

  • Disadvantages of XP

    • The dependency on customer being part of development team makes XP hard to adapt.

    • Most current projects are large and complex, which requires a large development team but communication will become extremely difficult for such projects.

    • The fact that the scope and timing is not well defined can result in a project that runs for ever.

    • Pair programming is not an easy thing to do. Some programmers prefer to code alone and find that pair programming slows them down.

    • Quality in XP does not get measured which is a problem for large projects because managers believe that keeping track of quality helps to produce a high quality software.

    • Some believe that some XP practices waist a lot of time without getting a significant gain such as refactoring.

    • XP requires the development team to be co-located and work at an open space work area which might not be comfortable for some developers.

› When to Use XP?

To put it all together, you should use XP when you have

  • A small project, probably with high risk.

  • Changing requirements.

  • A small development team willing to communicate and work together.

  • A customer willing to be part of the development team.
